# Skill 更新合并策略

用于在同一个 skill 上智能合并多条改进建议。

## 概览

当 improvement plan 中多项修改指向同一文件、甚至相同行时，需要：
- 检测冲突
- 按优先级解决
- 以正确顺序应用更新

## 基于优先级的合并

优先级顺序：
1. **High Priority**
2. **Medium Priority**
3. **Low Priority**

原因：
- 高优先级通常修的是基础问题
- 低优先级可能依赖高优先级修改
- 先做关键修复，避免返工

## 单文件合并策略

当多个改动命中同一文件时：

1. 先按文件分组
2. 再按行号从下往上处理
3. 如果命中同一行区间，则让更高优先级方案胜出
4. 把被跳过的冲突记录到报告中

## 跨文件合并策略

建议分三轮执行：

1. 所有高优先级改动
2. 所有中优先级改动
3. 所有低优先级改动

## 冲突类型

### 1. 同一行，不同建议

处理：优先级高者胜；同优先级则保留第一条并标记人工复核。

### 2. 行区间重叠

处理：高优先级胜出，低优先级跳过。

### 3. 同优先级冲突

处理：按出现顺序选第一条，写入冲突报告。

## 特殊情况

### 新文件创建

处理逻辑：
- 若文件不存在：创建
- 若已存在且内容一致：跳过
- 若已存在但内容不同：记为冲突

### 追加写入

如果无行号且目标文件已存在，可视为 append 操作，但要保证与已有内容不冲突。

### YAML Frontmatter 更新

frontmatter 应：
- 先 parse
- 更新具体字段
- 保留其余字段
- 最后验证 YAML 语法

## 合并后验证

应用完成后至少检查：
1. 目标文件存在
2. YAML 能解析
3. 建议内容确实写入
4. 没有意外副作用

## 最佳实践

1. 先备份
2. 高优先级优先
3. 底部到顶部处理，减少行偏移
4. 每处理完一个文件就做小验证
5. 所有决策写入更新报告

## 失败恢复

如果合并失败：
1. 立即停止
2. 记录失败点
3. 从备份恢复
4. 分析失败原因
5. 跳过问题改动后重试
